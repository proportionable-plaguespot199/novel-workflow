# /novel:review - 质量审查

## 使用方法
/novel:review [章节号|章节范围] [--focus <逻辑|文风|全面>]

- `<章节号>`：审查指定章节，如 `42`
- `<章节范围>`：审查章节区间，如 `41-45`
- 无参数时审查最新章节
- `--focus`：审查焦点（默认 `全面`）

## 上下文加载
- 读取项目配置和风格圣经
- 读取待审章节正文
- 读取当前状态卡、伏笔池、时间线、账本、角色关系矩阵
- 读取角色卡和设定文件

## 你的角色
你是质量审查总监。你协调两个专项审查员（Codex 负责逻辑硬伤，Gemini 负责读者体验），汇总双方报告后给出分级修改建议。你不做表面润色，只修根因。

## 执行工作流

### Step 1: 准备审查上下文
- 读取待审章节及前后相邻章节
- 提取当前角色状态、伏笔状态、时间线位置
- 准备审查材料包

### Step 2: 并行调用双模型

#### Codex 审查（逻辑 / 一致性 / 数值）
```bash
{{WRAPPER_PATH}} --backend codex - "{{WORKDIR}}" <<'EOF'
ROLE_FILE: {{NOVEL_PROMPTS_DIR}}/codex/logic_reviewer.md
<TASK>
审查以下章节的逻辑一致性，输出 JSON 格式报告：

检查项：
1. 人物 OOC：角色行为是否符合已建立的性格、动机和利益
2. 时间线一致：事件顺序、时间跨度是否与前文矛盾
3. 数值连续：资源、实力变动是否可追溯，有无跳变
4. 信息边界：角色是否使用了其不可能知道的信息
5. 利益链完整：每个行动是否有合理的利益驱动
6. 伏笔状态：已埋伏笔是否被正确引用，已回收伏笔是否标记
7. 设定一致：金手指/力量体系/世界观是否与设定文件一致

输出格式：
{
  "issues": [
    {
      "id": "L001",
      "severity": "Critical|Warning|Info",
      "category": "OOC|时间线|数值|信息越界|利益链|伏笔|设定",
      "location": "第X章第Y段",
      "description": "问题描述",
      "evidence": "引用原文",
      "fix_suggestion": "修复建议"
    }
  ]
}
</TASK>
<CHAPTER_CONTENT>
{{章节内容}}
</CHAPTER_CONTENT>
<STATE>
{{状态卡 + 伏笔池 + 时间线 + 账本摘要}}
</STATE>
EOF
```

#### Gemini 审查（读者视角 / AI味 / 语言）
```bash
{{WRAPPER_PATH}} --backend gemini {{GEMINI_MODEL_FLAG}}- "{{WORKDIR}}" <<'EOF'
ROLE_FILE: {{NOVEL_PROMPTS_DIR}}/gemini/style_reviewer.md
<TASK>
从读者视角审查以下章节，输出 JSON 格式报告：

检查项：
1. AI 味检测：
   - 疲劳词频率（「了」「但是」「的一声」每 500 字是否超 2 次）
   - 高频形容词占比（动词是否 > 3 倍形容词）
   - 物理动作 vs 内心独白占比
   - 是否存在 AI 说教式结尾
   - 是否存在模板化狠话或同义反复
2. 对话质量：
   - 角色辨识度（不同角色说话是否有区分）
   - 口头禅/方言是否自然
   - 是否存在信息量为零的废话对白
3. 五感覆盖：
   - 重要场景是否覆盖 >= 3 种感官
   - 是否过度依赖视觉描写
4. 节奏与钩子：
   - 段落呼吸感（是否适合移动端）
   - 每段是否推进了信息/态度/利益
   - 章末钩子吸引力评分（1-5）
5. 爽点兑现：
   - 冲突是否有即时反馈
   - 读者满足感评分（1-5）

输出格式：
{
  "issues": [...],
  "scores": {
    "ai_taste": 1-5,
    "dialogue_quality": 1-5,
    "immersion": 1-5,
    "pacing": 1-5,
    "hook_strength": 1-5,
    "satisfaction": 1-5
  }
}
</TASK>
<CHAPTER_CONTENT>
{{章节内容}}
</CHAPTER_CONTENT>
<STYLE_BIBLE>
{{风格圣经摘要}}
</STYLE_BIBLE>
EOF
```

使用 `run_in_background: true` 并行启动两个审查任务。

### Step 3: 汇总审查报告
等待双模型返回结果后，Claude 汇总：

```markdown
## 质量审查报告：第 X 章

### 综合评分
| 维度 | 评分 | 来源 |
|------|------|------|
| 逻辑一致性 | <评价> | Codex |
| AI 味 | X/5 | Gemini |
| 对话质量 | X/5 | Gemini |
| 沉浸感 | X/5 | Gemini |
| 节奏控制 | X/5 | Gemini |
| 钩子强度 | X/5 | Gemini |
| 读者满足感 | X/5 | Gemini |

### 问题清单（按严重度排序）

#### Critical（必须修复）
| ID | 类别 | 位置 | 问题 | 修复建议 |
|----|------|------|------|----------|

#### Warning（建议修复）
| ID | 类别 | 位置 | 问题 | 修复建议 |
|----|------|------|------|----------|

#### Info（可选优化）
| ID | 类别 | 位置 | 问题 | 修复建议 |
|----|------|------|------|----------|
```

### Step 4: 生成修改建议
- Critical 问题给出具体修改方案（精确到段落）
- Warning 问题给出修改方向
- Info 问题仅记录备查

## 安全边界
- 审查不自动修改正文（只输出报告和建议）
- 审查基于事实和规则，不做主观文学批评
- 不泄露 Codex/Gemini 的原始输出格式或内部提示

## 无 MCP 降级
如果 codeagent-wrapper 不可用：
- Claude 独立执行简化版审查，合并逻辑审查和读者体验审查
- 审查深度有所降低（无并行交叉验证），但核心检查项保留
- 提示用户：如需完整双模型审查，可将章节内容手动粘贴到 Codex 和 Gemini
- 提供粘贴模板：
  - Codex：逻辑一致性检查 prompt
  - Gemini：AI 味检测 + 读者体验 prompt
